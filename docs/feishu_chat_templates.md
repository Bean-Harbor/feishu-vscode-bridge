# Feishu Chat Templates

这份文档不是命令参考，而是给真实飞书聊天场景准备的“直接照发”模板。

目标只有一个：让你在飞书里尽量像在 Copilot Chat 里一样推进任务。

## 使用原则

- 先发目标，不要先发细节
- 如果需要连续推进，用 `执行计划` 或 `执行全部`
- 如果只是一次性操作，直接发自然语言命令即可
- 任务做完后，优先用追问把上下文接住，而不是重新描述一遍

## 模板 1：先看代码，再继续改

适合场景：你先想让桥接器读文件、看上下文，再决定怎么改。

发送顺序：

```text
读取 src/lib.rs 1-120
搜索 parse_intent 在 src
继续这个文件
```

也可以这样问：

```text
看上一步
```

建议用法：

- 第一条先把上下文拉出来
- 第二条确认关键符号在哪里
- 第三条回到刚才正在看的文件

## 模板 2：一步一步执行一个小任务

适合场景：你想让它稳一点，不要一口气跑完。

```text
执行计划 读取 src/bridge.rs 1-160; 搜索 ShowLastDiff 在 src; 查看 diff
继续
继续
```

什么时候适合：

- 你在探索代码
- 你不确定第二步以后是否要改方向
- 你希望每一步都看结果再决定

## 模板 3：一口气跑到失败点

适合场景：你知道任务链条，想先跑起来，失败再停。

```text
执行全部 运行测试 cargo test --lib; 查看 diff; 搜索 failure 在 src
```

如果停住了，接着发：

```text
为什么失败了
看上一步
继续
```

适合做：

- 测试 -> 看错误 -> 继续排查
- 代码搜索 -> 快速定位 -> 再决定是否修改

## 模板 4：补丁修改工作流

适合场景：你已经准备好 unified diff，想在飞书里直接改代码。

```text
应用补丁
diff --git a/src/lib.rs b/src/lib.rs
--- a/src/lib.rs
+++ b/src/lib.rs
@@ -1 +1 @@
-old
+new
```

补丁发完后，常用追问：

```text
看 diff
看文件列表
继续这个文件
撤回补丁
```

建议顺序：

1. 先 `看 diff`
2. 再 `看文件列表`
3. 如果要继续人工检查，发 `继续这个文件`
4. 如果补丁方向不对，发 `撤回补丁`

## 模板 5：审批型操作

适合场景：`git pull`、`git push`、`应用补丁`、shell 命令被策略拦住时。

你发：

```text
git pull
```

卡片里通常会看到：

- `确认继续`
- `取消这步`

确认后如果还想继续追上下文，可以再发：

```text
看上一步
继续刚才的任务
```

## 模板 6：失败后不要重说一遍任务

适合场景：计划执行到一半失败，你不想重新描述上下文。

直接发：

```text
为什么失败了
看上一步
继续刚才的任务
```

如果失败与文件有关，再补一条：

```text
继续这个文件
```

核心思路：

- 先拿失败原因
- 再看原始输出
- 再决定恢复计划还是回到文件

## 模板 7：代码修改后的复查

适合场景：你刚改完代码，想快速确认影响面。

```text
看 diff
看文件列表
运行测试 cargo test --lib
看上一步
```

这组模板适合做最小闭环：

- 看改了什么
- 看改到哪些文件
- 跑验证
- 看验证结果

## 模板 8：飞书里最顺手的一组短句

下面这组短句最适合高频追问，建议直接记住：

```text
继续
为什么失败了
看上一步
继续这个文件
看 diff
看文件列表
撤回补丁
```

如果你平时主要靠卡片按钮推进，可以把它们理解成按钮之外的文本等价物。

## 推荐聊天习惯

推荐顺序：

1. 先发任务目标
2. 再发执行模式：`执行计划` 或 `执行全部`
3. 出结果后优先用短追问，不要重复背景
4. 涉及补丁时，优先记住 `看 diff`、`看文件列表`、`撤回补丁`

不推荐：

- 每次失败后重新打一长段背景
- 刚看完文件又重新发一次完整路径描述
- 明明是补丁回滚，却重新手写第二个反向补丁

## 一条最小实战链路

如果你只记一套，就记这一组：

```text
执行全部 读取 src/bridge.rs 1-120; 搜索 build_plan_card 在 src; 查看 diff
为什么失败了
看上一步
继续这个文件
```

这组可以覆盖大多数“先看上下文，再继续追问”的开发对话。
