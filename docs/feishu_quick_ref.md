# Feishu Quick Ref

一页版速查卡，适合直接贴到飞书文档首页、群公告或内部 SOP。

## 最常用短句

```text
继续
为什么失败了
看上一步
继续这个文件
看 diff
看文件列表
撤回补丁
继续刚才的任务
```

## 两种推进方式

逐步推进：

```text
执行计划 命令1; 命令2; 命令3
继续
继续
```

连续推进：

```text
执行全部 命令1; 命令2; 命令3
```

选择原则：

- 不确定下一步怎么走，用 `执行计划`
- 想先跑起来，失败再停，用 `执行全部`

## 最小工作流

看代码：

```text
读取 src/lib.rs 1-120
搜索 parse_intent 在 src
继续这个文件
```

看改动：

```text
看 diff
看文件列表
```

看失败：

```text
为什么失败了
看上一步
```

回到任务：

```text
继续
继续刚才的任务
```

## 补丁场景

发补丁：

```text
应用补丁
diff --git a/src/lib.rs b/src/lib.rs
--- a/src/lib.rs
+++ b/src/lib.rs
@@ -1 +1 @@
-old
+new
```

补丁后立刻可用：

```text
看 diff
看文件列表
继续这个文件
撤回补丁
```

推荐顺序：

1. `看 diff`
2. `看文件列表`
3. `继续这个文件`
4. 方向不对就 `撤回补丁`

## 审批卡片按钮对照

卡片上看到：

- `继续下一步` = 发 `继续`
- `继续全部` = 发 `执行全部`
- `重试这步` = 发 `重新执行失败步骤`
- `确认继续` = 发 `批准`
- `取消这步` = 发 `拒绝`

追问按钮通常对应：

- `为什么失败了`
- `看上一步`
- `继续这个文件`
- `看 diff`
- `看文件列表`
- `撤回补丁`

## 最常见 4 条链路

链路 1：先看再改

```text
读取 src/bridge.rs 1-120
搜索 build_plan_card 在 src
继续这个文件
```

链路 2：跑到失败点

```text
执行全部 运行测试 cargo test --lib; 查看 diff; 搜索 error 在 src
为什么失败了
看上一步
```

链路 3：补丁后复查

```text
应用补丁
<diff>
看 diff
看文件列表
运行测试 cargo test --lib
```

链路 4：审批后接着追问

```text
git pull
批准
看上一步
继续刚才的任务
```

## 使用原则

- 先发目标，再发执行模式
- 出结果后优先用短追问，不要重打一遍背景
- 涉及补丁时，优先记住 `看 diff`、`看文件列表`、`撤回补丁`
- 想稳一点就走 `执行计划`，想快一点就走 `执行全部`
